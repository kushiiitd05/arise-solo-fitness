## Plan: 01-01 — API Route Safety Fixes

### Context
Two API routes have critical flaws:
- `src/app/api/xp/award/route.ts` uses `.single()` on both the users and user_stats queries. When either row doesn't exist (e.g., stats not yet seeded), Supabase returns a 406 PGRST116 error and the entire endpoint crashes.
- `src/app/api/user/route.ts` has a `getUserId()` function that accepts a `userId` URL query param as a fallback to Bearer token. This means any unauthenticated caller can read or modify any user's data by passing `?userId=<uuid>`.

### Files to Modify
- `src/app/api/xp/award/route.ts`
- `src/app/api/user/route.ts`

### Tasks

#### Task 1: Fix .single() → .maybeSingle() in /api/xp/award
In `src/app/api/xp/award/route.ts`:
- Change the user fetch from `.single()` to `.maybeSingle()`
- Change the user_stats fetch from `.single()` to `.maybeSingle()`
- Add null check after each: if user is null return 404, if stats is null skip the stats update (don't crash)

#### Task 2: Remove userId query param auth in /api/user
In `src/app/api/user/route.ts`:
- Remove the `getUserId()` helper that reads from query params
- For GET and PATCH: extract userId from the Authorization Bearer header only
- If no valid Bearer token → return 401
- The POST handler (user creation/upsert) can remain as-is since it receives full body with id

#### Task 3: Verify build compiles
Run `npm run build` or `npx tsc --noEmit` to confirm TypeScript types are valid after changes.

### Success Criteria
- [ ] `/api/xp/award` returns 404 (not 406/500) when user not found
- [ ] `/api/user` GET/PATCH returns 401 when called without Authorization header
- [ ] TypeScript compilation passes with no new errors
